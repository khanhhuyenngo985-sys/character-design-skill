# 讨论格式规范

## 讨论触发时机

每个关键节点（Phase 2, 3-4, 5, 7, 8）必须进入讨论模式。

## 讨论发起格式

```markdown
## 讨论：[话题]

### 背景
[2-3句话说明这个决策的背景和重要性]

### 我的推荐
**[推荐方案]** — [一句话理由]

### 其他选项
| 方案 | 优点 | 缺点 |
|------|------|------|
| A | ... | ... |
| B | ... | ... |

### 我的理由
[2-3句话解释为什么推荐这个方案，引用上游简报或知识库]

### 你的意见？
[等待用户回复]
```

## 用户回复处理

### 用户同意推荐方案
```markdown
收到，同意。
记录决策：[方案名称] — [用户补充，如有]
```

### 用户要求修改
```markdown
收到，你的修改意见是：
1. [修改点]
2. [修改点]
正在调整...
```

### 用户否决所有选项，提出新方向
```markdown
收到，收到新方向建议。
正在基于 [新方向] 重新设计方案...
```

### 双方意见僵持
```markdown
记录分歧：
- AI 推荐：[方案A]，理由：[...]
- 用户倾向：[方案B]，理由：[...]
输出双版本：[方案A版本] 和 [方案B版本]
请用户最终拍板。
```

## 讨论结束标准

- 用户明确说"可以"、"同意"、"就这么办"
- 或者用户给出最终决策
- 才认为讨论结束，记录到 decision_log

## 讨论频率控制

- 每个关键节点至少讨论一轮
- 如果 AI 有强烈推荐，必须给出理由
- 用户可以随时说"继续"跳过讨论
- AI 不得连续3次催促用户
